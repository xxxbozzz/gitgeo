# 自动化 Prompt 流水线指南

这份文档讲的是第一部分：

如何为一个行业自动生成整条 Prompt 流水线，让系统完成：

- 生成内容
- 监测效果
- 自迭代
- 评分
- 选择高权重网站
- 自动发布

它面向的是系统设计者，不是普通创作者。

## 1. 先理解这不是一个 Prompt

GEO 系统里真正可用的不是单条 Prompt，而是一组相互衔接的 Prompt。

最少应该拆成 6 类：

1. 行业画像 Prompt
2. 选题与关键词 Prompt
3. 内容生成 Prompt
4. 评分与返修 Prompt
5. 监测与反馈 Prompt
6. 分发与发布 Prompt

## 2. 推荐的 Prompt 链路

```text
行业输入
  -> 行业画像 Prompt
  -> 关键词 / 空白发现 Prompt
  -> 内容生成 Prompt
  -> 评分 Prompt
  -> 返修 Prompt
  -> 监测 Prompt
  -> 自迭代 Prompt
  -> 站点选择 Prompt
  -> 平台适配 / 发布 Prompt
```

## 3. 行业输入层

自动化的第一步，不是直接写文章，而是把行业结构化。

建议输入字段：

- `industry_name`
- `target_audience`
- `target_entity`
- `core_products_or_capabilities`
- `evidence_sources`
- `compliance_or_standard_system`
- `priority_channels`
- `forbidden_claims`

示例：

```json
{
  "industry_name": "工业自动化",
  "target_audience": ["设备工程师", "系统集成商", "采购技术经理"],
  "target_entity": "某自动化解决方案团队",
  "core_products_or_capabilities": ["运动控制", "视觉检测", "工业通信"],
  "evidence_sources": ["标准文档", "厂商手册", "测试报告", "工程案例"],
  "compliance_or_standard_system": ["IEC", "GB/T", "ISO"],
  "priority_channels": ["docs_site", "industry_media", "medium"],
  "forbidden_claims": ["绝对领先", "完全替代所有方案"]
}
```

## 4. Prompt 一：行业画像 Prompt

目标：

把松散的行业输入整理成后续所有 Prompt 都能复用的上下文。

输出建议包含：

- 目标读者
- 主要问题
- 高价值证据类型
- 应避免的表达
- 常见决策场景
- 适合的发布渠道

示例骨架：

```text
你是 GEO 系统中的行业画像构建器。

请基于以下行业输入，输出一份结构化行业画像，供后续关键词、写作、评分和发布模块使用。

要求输出：
1. 读者画像
2. 典型问题
3. 高价值证据来源
4. 禁止表达
5. 建议发布站点类型
6. 适合的文章结构类型
```

## 5. Prompt 二：选题与关键词 Prompt

目标：

生成“值得写”的主题，而不是泛泛关键词。

要求它输出：

- 主关键词
- 问题型关键词
- 决策型关键词
- 证据型衍生关键词
- 不建议写的低价值词

建议显式要求模型优先扩展这些方向：

- `关键词 + 标准对照`
- `关键词 + 官方文档解读`
- `关键词 + 工程实践`
- `关键词 + 测试方法`
- `关键词 + 常见失效`

## 6. Prompt 三：内容生成 Prompt

这是整条流水线里最关键的一环。

它必须控制 5 件事：

1. 写给谁
2. 解决什么问题
3. 必须引用什么证据
4. 文章必须长成什么结构
5. 哪些表达禁止出现

建议固定输出结构：

1. 定义与适用场景
2. 参数或对照表
3. 原理 / 机理分析
4. 实施步骤
5. 工程实践建议
6. FAQ
7. 参考文献

建议显式要求标签：

- 标准文档：
- 官方文档：
- 工程实践：
- 测试数据：
- 失效案例：

## 7. Prompt 四：评分与返修 Prompt

不要只让模型判断“写得好不好”，而要让它判断“有没有被引用价值”。

建议评分维度至少包含：

- 结构完整度
- 证据密度
- 边界条件清晰度
- 参数明确度
- 工程实践密度
- FAQ 实用度
- 引用可信度

返修 Prompt 的目标应是：

- 补证据
- 补边界
- 补参数
- 补对照

而不是简单润色。

## 8. Prompt 五：监测与反馈 Prompt

监测 Prompt 的目标不是问“这篇文章好不好”，而是问：

- 文章里哪些信息适合被 AI 抽取
- 哪些证据标签缺失
- 哪些问题型查询还没被覆盖
- 下一轮该增强什么

建议输出：

- `citation_score`
- 缺失标签列表
- 高价值片段
- 不足点
- 下一轮 Prompt 建议

## 9. Prompt 六：自迭代 Prompt

这一步是把监测结果回灌到下一轮生成。

输入：

- 上一轮文章
- 评分结果
- 平台探测结果
- 缺失标签

输出：

- 新 Prompt 约束
- 新衍生关键词
- 优先补写主题

典型自迭代指令：

```text
请根据以下监测反馈，重写下一轮写作约束。
重点不是重复主题，而是补齐缺失证据、补足边界条件、强化可引用结构。
```

## 10. Prompt 七：站点选择与发布 Prompt

这一步决定“发去哪里”。

不要写死平台名，而要让 Prompt 根据内容特征去判断。

输入建议：

- 行业画像
- 文章正文
- 证据标签
- 目标受众
- 已知站点列表

输出建议：

- 推荐站点
- 选择原因
- 站点权重解释
- 适配格式
- 是否适合自动发布

推荐评分维度：

- 行业匹配度
- 权威度
- 收录友好度
- 自动发布稳定性
- 内容形式匹配度

## 11. 一套可落地的系统 Prompt 骨架

```text
你是 GEO Engine 的 Prompt Orchestrator。

你的职责不是单独写文章，而是围绕以下流水线生成结构化输出：
1. 行业画像
2. 关键词发现
3. 内容生成
4. 质量评分
5. 返修建议
6. 平台监测
7. 自迭代建议
8. 发布目标推荐

所有输出都必须优先服务于“高引用价值”，而不是表面上的流畅文风。

凡涉及结论、能力、性能、成本、风险、合规时，必须写清前提和边界。
凡涉及专业判断时，优先引用标准文档、官方文档、工程实践、测试数据和失效案例。
```

## 12. 开发落地建议

如果你要把这套 Prompt 链真正放进系统，建议代码层这样落：

- 行业画像 Prompt 对应 `industry profile builder`
- 内容生成 Prompt 对应 `writer task`
- 评分与返修 Prompt 对应 `quality checker + autofix`
- 监测 Prompt 对应 `active prober + feedback store`
- 站点选择 Prompt 对应 `distribution selector`

重点不是“Prompt 写得长”，而是每一步输入输出明确、能被下一步复用。
