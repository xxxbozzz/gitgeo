# GEO Engine 系统结构总览

这个开源项目的重点不是某个特定行业案例，而是整套 GEO 系统如何运转。

如果只看一条主线，可以把系统理解成 7 个层次：

1. 关键词与选题层
2. 证据采集层
3. Prompt 编排层
4. 内容生成与返修层
5. 质量评分与引用信号层
6. 监测与自迭代层
7. 发布与分发层

## 1. 主流程

```text
关键词发现
  -> 证据采集
  -> Prompt 组装
  -> 内容生成
  -> 质量检查 / 自动返修
  -> 引用信号分析
  -> AI 平台主动探测
  -> 反馈写回
  -> 衍生关键词扩展
  -> 发布目标选择
  -> 渠道适配与发布
```

## 2. 目录与职责

### `batch_generator.py`

系统主入口。

负责串联：

- 关键词消费
- 内容生成
- 质检
- 反馈写回
- 后续处理

### `core/`

系统核心逻辑层。

重点文件：

- `tasks.py`
  定义生成任务和 Prompt 注入逻辑。
- `tools.py`
  定义搜索、知识库、发布等工具。
- `active_prober.py`
  主动探测 AI 平台可见性。
- `prompt_optimizer.py`
  从评分和监测结果中提炼下一轮 Prompt 指导。
- `feedback_store.py`
  保存探测结果、反馈摘要和关键词级信号。
- `distribution_selector.py`
  为文章选择更合适的发布目标。
- `site_registry.py`
  管理站点元信息、权重与发布能力声明。
- `publisher_adapters.py`
  把具体平台的发布实现封装成统一适配层。

### `backend/`

后端 API 层。

负责：

- 文章、运行、系统状态等接口
- 统一向前端暴露数据
- 把核心能力变成可操作的服务

### `frontend_v2/`

新控制台。

负责：

- 系统状态展示
- 内容管理
- 运行视图
- 发布视图
- 后续反馈中心的承载

### `dashboard/`

旧版 Streamlit 控制台与原型。

目前更适合作为过渡界面和历史控制台。

### `database/`

数据库结构与反馈表定义。

重点是：

- 文章和关键词主表
- 反馈与探测表
- 后续可扩展的发布指标表

## 3. 四个关键闭环

### 闭环一：选题闭环

目标是找到值得写、值得监测、值得继续扩展的主题。

输入：

- 关键词
- 行业问题
- 平台回答空白

输出：

- 待生成主题
- 衍生关键词
- 优先级

### 闭环二：内容闭环

目标是让文章不仅能生成，还能被判定为“像资料”。

输入：

- 结构模板
- 证据要求
- 行业约束

输出：

- 文章正文
- 质量分
- 返修结果

### 闭环三：引用闭环

目标是分析内容是否具备高引用价值。

系统重点看这些信号：

- 标准文档
- 官方文档
- 工程实践
- 测试数据
- 失效案例

输出：

- `citation_score`
- 缺失标签
- 下一轮 Prompt 加强建议

### 闭环四：分发闭环

目标不是“群发更多渠道”，而是优先命中高权重、适合当前内容的站点。

系统需要结合：

- 行业画像
- 证据类型
- 站点权重
- 发布稳定性

输出：

- 推荐发布目标
- 站点评分
- 适配后的发布动作

## 4. 开源版最值得理解的能力

这个项目当前最有价值的不是某篇文章，而是这些结构：

- Prompt 可以被拆解、组合和自迭代
- 评分不只看文风，还看引用价值
- 监测结果会回流到下一轮生成
- 发布层已经朝“适配器 + 站点选择器”演进
